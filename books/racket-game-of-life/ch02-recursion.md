#### 2.1 デザインレシピとテンプレート（章の定型）

第1章では「小さな関数」と `check-expect` を書きました。第2章からは、**毎回同じ手順に沿って**関数を設計します。これが HtDP / howtocode の **デザインレシピ**です。いきなり完成コードを書くのではなく、データ・例・骨組みを先に置き、途中で迷わないようにします。

howtocode ではテンプレートを **outline（アウトライン）** とも呼びます。プログラムを書く前に、使う部品（引数、フィールド、場合分け）をコメントや紙に書き出しておく作業です。慣れるまでは、頭の中だけで済ませずに書き出すとよいです。

##### なぜ「定型」が必要か

- 再帰やリストは、止まる条件や場合分けを忘れると、エラーや無限ループになりやすい  
- データの「どんな場合があるか」と、関数の `cond` の枝を**同じ個数・同じ意味で対応**させると、抜け漏れが減る  
- ライフゲームの「生存リストを先頭から見る」「近傍を数える」も、同じ手順の繰り返しになる  

第2章で身につける手順は、第4章の `next-generation` でもそのまま使います。

##### パートA: データの設計（何を表すか）

関数を書く前に、**「扱うデータの決まり」**を先に決めます。howtocode では次の4つです。高校で数学の「集合の定義」や「場合分け」に近い作業だと思ってください。

**1. Data Description（データの記述）**

「このデータは何という名前で、どんな形を取りうるか」を文章（コメント）で書きます。

- 名前を付ける（例: `ListOfNumber`）  
- **取りうる形をすべて列挙する**（「次のいずれか」）  
- リストなら典型的に2つ:  
  - 何も入っていない → `empty`  
  - 先頭が1つあり、残りも同じ種類のリスト → `(cons 要素 残りのリスト)`  

例:

```text
ListOfNumber は次のどちらかである。
  - empty
  - (cons Number ListOfNumber)
```

ここで「どちらか」と書いておかないと、あとで関数を書くとき「空のときどうする？」が抜けます。数学で「自然数は 0 か、ある自然数に 1 を足したもの」と定義するのと、同じ発想です。

**2. Interpretation（解釈）**

そのデータが、**問題の世界で何を意味するか**を日本語で書きます。同じプログラム上の形でも、意味が違えば正しい使い方が変わります。

- 悪い例: 「リスト」だけ（何のリストか分からない）  
- よい例: 「テスト用の整数が並んだ列」「ライフ盤で今生きているセルの座標 (x, y) が並んだ列」  

解釈はコメントに `interp.` と書いて残すことが多いです。後から自分や読者が見たとき、「このリストに死んだセルは入れない」などの約束が分かります。

**3. Data Examples（データの具体例）**

定義した形に当てはまる**実際の値**を2〜3個、プログラムに書きます。

リストなら次を揃えると安全です。

- **空**: `empty`（長さ 0。境界のケース）  
- **1個**: `(cons 99 empty)`  
- **2個以上**: `(cons 4 (cons 6 empty))`  

「長さ2以上」の例があると、「先頭を処理して、残りに同じ処理をする」再帰の動きを追えます。howtocode も、リストを引数にする関数では長さ2以上のテストを勧めています。

例はあとで `check-expect` の入力にも使います。「定義は書いたが例が無い」状態だと、動くかどうか確かめにくいです。

**4. このデータを処理する template（テンプレート）**

「この型の値を1つ受け取って何かを返す関数」が、**必ず通る道筋**を、まだ答えを埋めずに書きます。

- データの「場合」が2つなら、`cond` の枝も2つ  
- 各枝で、その場合に使える部品だけを書く  
  - 空なら: 空だと分かることだけ使える  
  - `cons` なら: `first`（先頭）と `rest`（残り）が使える。残りはまた同じ型なので、同じ関数を `rest` に書ける枠を残す  

答えの部分は **プレースホルダ `...` / `(...)`** で空けておきます（次の「`...` について」で詳しく説明します）。テンプレートは「骨組み」であり、この段階では完成した答えを出す必要はありません。

##### パートB: 関数の設計（どう計算するか）

データの決まりができたら、**関数を1つずつ**次の5段で書きます。

1. **Signature, purpose, stub**  
   - **署名（signature）**: 何を受け取り何を返すか。コメントで書く方法と、後述の `(: …)` でコードに書く方法がある  
   - **目的（purpose）**: 日本語1文。例: 「リストの要素をすべて足した値を返す」  
   - **stub**: とりあえず定義だけ置き、本体は仮（いつも `0` を返すなど）。先にテストを書けるようにするため  

2. **Examples（`check-expect`）**  
   「この入力ならこの出力」を2つ以上書く。これが仕様になる。実装を後から直しても、仕様が守れているかを機械的に確認できる。

3. **Template（データから）**  
   パートAの4で書いた骨を、この関数用にコピーする。リストなら `empty?` / `first` / `rest` と、残りへの自己呼び出しの枠がすでに並んでいる。

4. **Code body（本体）**  
   テンプレートの空き（`...`）を、本物の式で埋める。

5. **Test, review, refactor**  
   Run して `check-expect` が通るか確認する。通ったら目的文と例がまだ正しいか見直し、重複があれば整理する。

##### `...` / `(...)` について（BSL が許す「仮置き」）

テンプレートに出てくる **`...`** や **`(...)`** は、「あとで埋める場所」を表す記号です。なぜ文法（Syntax）の本編より前に、こんな記号が特別扱いされているのかを説明します。

- Racket の初心者向け言語（BSL）は、単に「動くプログラムを書く」ためだけでなく、**プログラムを段階的に設計する手順（デザインレシピ）**を学ぶために作られています。  
- 設計の途中（データ定義から関数の骨組み＝テンプレートを作る段階）では、まだ具体的な計算の中身は決まっていません。  
- 普通のプログラミング言語では、書きかけの未完成コード（空欄や未定義の名前）を書くと **文法エラー** になり、Run して途中経過を確かめられません。  
- そこで BSL では、`..` や `...`（点の個数は教材によって数種類ある）を **「エラーにしないための特別な変数（プレースホルダ）」** として言語にあらかじめ登録してあります。テンプレートを Run しても、設計の途中だと分かる結果になり、文法エラーで作業が止まらないようにしてあります。  

まとめ:

- `...` / `(...)` は「ここに後で本物の式を書く」印である  
- デザインレシピの途中で骨組みを試すために、BSL が**意図的に文法として許している**  
- 関数が完成したら、プレースホルダは `0` や `(+ (first lon) (sum-list (rest lon)))` のような**実際の式**に置き換える  
- 提出用・付属の完成コード（`code/ch02-recursion.rkt`）には、通常プレースホルダは残さない  

公式: [Beginning Student](https://docs.racket-lang.org/htdp-langs/beginner.html)（Template Variables 付近）

##### テンプレートと本体の関係

```text
  データ定義  →  テンプレート（cond の枝・first/rest・空き ... ）
                      ↓
  check-expect  →  本体で ... を式に置換  →  再テスト
```

以降の節では、この手順を **ListOfNumber → 再帰関数 → 区分 → ListOfPosn** の順で具体化します。

#### 2.2 データ定義: `ListOfNumber`

まず「数のリスト」をデータとして固定します。定義の中に同じ型名 `ListOfNumber` がもう一度出てくる（自己参照）ことが、再帰の出発点です。

```racket
;; ListOfNumber is one of:          ; どちらか一方
;;  - empty                         ; 要素が0個
;;  - (cons Number ListOfNumber)    ; 先頭が数、残りがまた ListOfNumber
;; interp. 数のリスト（合計・長さなどの練習用）

(define LON0 empty)                   ; 空の例（境界）
(define LON2 (cons 4 (cons 6 empty))) ; 2要素: 先頭4、残り (6)
```

- `cons` は「先頭を1つ足した新しいリスト」を作る。元のリストは書き換えない  
- 2要素の例があると、「先頭を処理して残りに同じ関数」の流れを追いやすい  
- 付属コードでは `LON1`（1要素）や `LON3`（3要素）も用意している  

##### `cond` の文法（BSL）

テンプレートや再帰では `cond` をよく使います。Beginning Student における式の主な形は、公式ドキュメントでは次のように整理されています（抜粋・説明用）:

```text
式 expr は、おおむね次のいずれか:

  (名前 式 式 ...)              関数呼び出し・演算
  (cond [式 式] ... [式 式])    複数の「条件 → 結果」
  (cond [式 式] ... [else 式])  最後を else にした形
  (if 式 式 式)                 真のとき / 偽のときの二択
  (and 式 式 式 ...)            すべて真か
  (or 式 式 式 ...)             どれかが真か
```

出典: [Beginning Student — Syntax](https://docs.racket-lang.org/htdp-langs/beginner.html)（Grammar / Common Syntaxes 付近）。

**`cond` の読み方**

- 形: `(cond [質問1 答え1] [質問2 答え2] ... [else 答え])`  
- **上から順に**質問を評価し、最初に真になった枝の「答え」が全体の値になる  
- 質問は `#true` / `#false` になる式（例: `(empty? lon)`、`(= n 0)`）  
- `else` は「ここまでどれも真でなかったとき」用の最後の枝（任意だがよく使う）  
- 角括弧 `[ ]` は「1本の枝」を囲む。丸括弧でも書けるが、教材では `[質問 答え]` が多い  

`if` は枝がちょうど2つ（真のとき・偽のとき）のとき向きです。枝が3つ以上、または「空 / 非空」「0 / 1 / それ以外」のように増えるときは `cond` の方が読みやすいです。

このデータに対応する**関数テンプレート**は、場合が2つなので `cond` も2枝です。

```racket
;; list-of-number-temp: まだ「何を返すか」は未定。骨だけ示す
;; （完成ファイル code/ch02-recursion.rkt には置かない）
(define (list-of-number-temp lon)
  (cond
    [(empty? lon) (...)]    ; 質問: 空か？ 真なら基本ケース
    [else
     (... (first lon)       ; 先頭の数をどう使うか（後で埋める）
          (list-of-number-temp (rest lon)))]))
          ; 残り（同じ型）に同じ関数 → 自己参照＝再帰の枠
```

- `empty?` 枝を忘れると、空リストで止まらない  
- `else` 枝では「先頭への処理」と「残りへの再帰」の両方を式に含めるのが普通  
- `...` は前節のとおり BSL のプレースホルダ。次節では実際の式に置き換える  

##### シグネチャ（`: `）— コメントだけではない書き方

**引数と返り値（ステップ）**

1. 関数を呼ぶときは、括弧の中に値を渡す。渡す値が **引数** である。  
   例: `(square-of 8)` の `8` が引数。  
2. 関数は計算したあと、必ず何か値を返す。返ってくる値が **返り値** である。  
   例: `(square-of 8)` の返り値は `64`。  
3. 引数が複数あることもある。`(average 2 8)` なら第1引数が `2`、第2引数が `8`。  
4. **署名（シグネチャ）**は、「どんな型の引数を、何個受け取り、どんな型の返り値を返すか」という約束を書いたもの。  
5. 矢印 `->` の書き方では、だいたい次のように読む。  
   - `(Number -> Number)` … 数を1つ受け取り、数を返す  
   - `(Number Number -> Number)` … 数を2つ受け取り、数を返す  
   - `(Boolean Number -> Boolean)` … 真偽と数を受け取り、真偽を返す  

デザインレシピの「署名」は、コメントで `(ListOfNumber -> Number)` と書く方法もありますが、**コードの一部として書く**こともできます。

構文:

```racket
(: name signature-form)
```

意味:

- `signature-form` で指定したシグネチャを、`name` の定義に付ける  
- プログラムのどこかに `name` の `define` がなければならない  
- シグネチャは「この名前がどういう型の値か／関数なら引数と結果の型」を表す  
- 関数にシグネチャが付いていると、DrRacket は**その約束どおりに使っているか**を検査し、テスト結果とともにシグネチャ違反を出せる  
- シグネチャ形式は、シグネチャ宣言やシグネチャ式の内側でのみ働く特別な構文である  

正しい例:

```racket
(: age Integer)
(define age 42)

(: area-of-square (Number -> Number))
(define (area-of-square len)
  (sqr len))
```

約束に合わない例:

```racket
(: age Integer)
(define age "fortytwo")
```

この場合、実行するとシグネチャ違反が報告されます（`Integer` のはずが文字列 `"fortytwo"` になった、など）。

詳細は [Beginning Student — Signatures](https://docs.racket-lang.org/htdp-langs/beginner.html) を参照してください。本の付属コードはコメント署名と `check-expect` を中心にしつつ、本文では `(: …)` も示します。

#### 2.3 単純再帰の例

「より小さい同じ問題」に帰着させるのが再帰です。数の階乗は引数 `n` を減らす再帰、リストは `rest` で短くする再帰です。どちらも**止まる条件（基本ケース）**が必須です。

次の例では、各関数に **`(: …)` シグネチャ**を付けています。

```racket
;; 非負整数 n の階乗。n! = n * (n-1) * … * 1（0! と 1! は 1）
(: factorial (Number -> Number))
(define (factorial n)
  (cond
    [(<= n 1) 1]                          ; 止まる条件（基本ケース）
    [else (* n (factorial (- n 1)))]))    ; n を1減らした答えに n をかける

;; リストの要素数。空なら0、そうでなければ 1 + 残りの長さ
(: my-length ((ListOf Number) -> Number))
(define (my-length lon)
  (cond
    [(empty? lon) 0]                      ; テンプレの empty 枝
    [else (+ 1 (my-length (rest lon)))])) ; 先頭は「1個分」、残りを再帰

;; 要素の合計。空なら0、そうでなければ 先頭 + 残りの合計
(: sum-list ((ListOf Number) -> Number))
(define (sum-list lon)
  (cond
    [(empty? lon) 0]
    [else (+ (first lon) (sum-list (rest lon)))]))
```

- `my-length` / `sum-list` は 2.2 のテンプレと枝が同じで、埋めた式だけが違う  
- BSL には `map` / `filter` / `lambda` が無いので、リスト処理は**この構造的再帰**が中心になる  
- Intermediate 以降で高階関数を学んだら、同じ処理を短く書けることもある（予告）  

付属コードでは `fib`（フィボナッチ）や `my-member?`（ある数がリストに含まれるか）も同じ考え方で定義し、`check-expect` で固定しています。

#### 2.4 `cond` による区分（enum / interval）

リスト以外でも、**有限個の場合**や**数の範囲**で意味が変わるデータは、`cond` の枝をデータに合わせて増やします。ライフの近傍数は 0〜8 の整数なので、ラベル付けの練習に使えます。

```racket
;; 近傍の生存数 n を、ざっくりした状況ラベルに変換する（教育用）
(: neighbor-label (Number -> String))
(define (neighbor-label n)
  (cond
    [(= n 0) "empty"]                         ; 周りに誰もいない
    [(= n 1) "lonely"]                        ; 1人だけ（過疎寄り）
    [(or (= n 2) (= n 3)) "stable-or-birth"]  ; B3/S23 で生き残る/生まれる帯
    [else "overcrowd"]))                      ; 4以上など、過密寄り
```

- 上から順に最初の真の枝が使われる。枝の順序を変えると結果が変わることがある  
- 第1章の `survives?` / `births?` は真偽値を返す。ここでは同じ数を**文字列ラベル**で見ている  
- 本線の生死判定は引き続き「2 または 3」の規則。ラベルは理解のための補助  

howtocode の enum / interval も、「場合の個数＝ `cond` の枝の個数」という点ではリストのテンプレートと同じです。

#### 2.5 `ListOfPosn` と近傍

盤面の生存セルは、第1章どおり `posn` のリストで表せます。数のリストと同じ自己参照で、要素の型が `Number` ではなく `Posn` になっただけです。

```racket
;; ListOfPosn is one of:
;;  - empty
;;  - (cons Posn ListOfPosn)
;; interp. 生存セルの座標の列（死セルはリストに載せない）

;; 各セルの x 座標だけを取り出したリストを返す
(: xs-of ((ListOf Posn) -> (ListOf Number)))
(define (xs-of cells)
  (cond
    [(empty? cells) empty]                    ; 空なら空リスト
    [else (cons (posn-x (first cells))        ; 先頭の x を結果の先頭に
                (xs-of (rest cells)))]))      ; 残りの x 列を再帰で作る
```

- `cons` で「変換した先頭」と「変換した残り」をつなぐと、長さが同じ新しいリストになる  
- 「各要素を変換したリストを作る」処理の基本形（BSL では `map` の代わりにこう書く）  
- 近傍は第1章と同じく、`shift-cell` でずらした 8 点を `list` で並べる（`code/ch02-recursion.rkt` の `cell-neighbors`）  

第4章では、このリストを先頭から歩きながら「生きているか」「近傍が何個か」「次世代に残すか」を、同じテンプレート再帰で組み立てます。

#### 2.6 まとめ

この章で使った用語を、それぞれ「何か・何のためか・どこに書くか」が分かるように整理します。

**データ定義**

- **何か**: 「このプログラムが扱う値は、どんな形を取りうるか」を決めた約束。多くの場合コメントで書き、「次のいずれか（one of）」と場合を列挙する。  
- **何のためか**: 関数を書くとき、場合分けの本数と意味の根拠になる。定義に無い場合を `cond` に足したり、定義にある場合を忘れたりしないため。  
- **どこに書くか**: 関数より前。名前（例: `ListOfNumber`）、場合の列挙、解釈（`interp.`）、具体例（`define`）をセットにする。

**テンプレート**

- **何か**: あるデータ定義の値を引数に取る関数が、**必ず使う場合分けと部品**を先に並べた下書き。リストなら `empty?` の枝と `else` の枝、`first` / `rest`、残りへの自己呼び出しの枠など。  
- **何のためか**: いきなり完成コードを書かず、データに沿った骨組みを確認してから中身を埋めるため。  
- **どこに書くか**: 設計メモや学習用の下書き。答えの位置には `...` を置き、完成したら本物の式に置換する。提出用の完成ファイルには通常残さない。

**`check-expect`**

- **何か**: 「この入力を与えたら、この出力になるはずだ」と書いたテスト式。  
- **何のためか**: 仕様をコードとして固定し、Run のたびに自動で合否を見るため。実装を直したあとで壊れていないかも分かる。  
- **どこに書くか**: 関数の直前または直後。デザインレシピでは本体を埋める前に書くことが多い。

**シグネチャ（`: `）**

- **何か**: 「この名前はどういう型か／関数なら引数と結果の型」を、`(: 名前 シグネチャ形式)` で付けた宣言。コメント署名のコード版。  
- **何のためか**: DrRacket が型の約束違反を検査し、テストと一緒に報告するため。  
- **どこに書くか**: 対応する `define` の近く（直前が分かりやすい）。

**構造的再帰**

- **何か**: データの定義の形に合わせて、問題を小さくしてから自分自身を呼ぶ書き方。  
- **具体例**: リストなら `rest` に対して再帰する（リストが短くなる）。数の階乗なら `(- n 1)` に対して再帰する（数が小さくなる）。  
- **何のためか**: リスト全体や数 `n` を一括で処理するとき、BSL で確実に止まると分かる手順を得るため。  
- **「構造的」の意味**: 再帰の仕方（どこを小さくするか）が、データ定義からそのまま読み取れる、という意味。

高階関数（`map` など）に頼らなくても、データ定義・テンプレート・`check-expect`・（任意で）シグネチャがあれば、長いプログラムも段階的に書ける。次の第3章ではグリッド表現、第4章では `next-generation` にこの手順を接続する。
