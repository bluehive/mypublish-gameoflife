#### 3.1 なぜ「グリッド」をデータで決めるか

画面の升目そのものをいきなり描く前に、**プログラムが扱う値の形**を決めます。

ここで言う「第2章のデザインレシピと同じ」とは、次の**同じ順番**で進める、という意味です。

1. **データ定義** — 「どんな形の値を扱うか」をコメントで決める  
2. **解釈（interp.）** — それが盤の何を意味するかを日本語で書く  
3. **具体例** — `define` で実際の値を2〜3個置く  
4. **テンプレート** — その形を処理する関数の骨（`cond` など）を `...` で書く  
5. **スタブ** — 名前と引数だけ揃えた仮の関数（中身はダミー）  
6. **`check-expect`** — 「この入力ならこの出力」を先に書く  
7. **本体** — `...` やダミーを本物の式に置き換える  

第1–2章では数やリストの長さなどでこの流れを練習しました。本章では同じ流れを、**ライフの盤面**に当てはめます。

- 形が決まれば、関数の `cond` の枝や使う部品（`first` / `rest` など）もほぼ決まる  
- ライフでは「どこが生きているか」が中心なので、その表し方を選ぶ  

本章では表し方を **2通り**紹介します。

- **本線（A）** … この本の第4章以降で**ずっと使う**やり方。学習のメインルート  
- **理解用（B）** … 説明や小さな例のために使う補助。本線ではない  

**A. 生存セルだけを並べる（スパース）**

- データ: `ListOfPosn`（第1–2章と同じ）  
- 生きている座標だけをリストに入れる。死は「リストに無い」で表す  
- 空の盤面は `empty`  
- **スパース** … 「すき間が多い」「全部は書かない」という意味。死マスを0で埋めない  

**B. 升目を全部持つ（密グリッド）**

- データ: 行のリスト。各行は 0/1 のリスト  
- すべてのマスを書くので、小さな図をそのままコードにしやすい  
- **密** … 「すき間なく全部書く」という意味。3×3 なら9個の 0/1 が必ずある  

どちらも「正しさ」は `check-expect` で固定します。

#### 3.2 データ定義 A: `ListOfPosn`（本線）

ここでは第2章で学んだ次の内容を、**盤面用**にそのまま使います。

- リストは `empty` か `(cons 先頭 残り)`  
- 残りも同じ型のリスト（自己参照）  
- 処理は `empty?` / `first` / `rest` のテンプレート  

第2章では要素が「数」でした。本章の本線では要素が **`posn`（座標）** です。

```racket
;; ListOfPosn is one of:
;;  - empty
;;  - (cons Posn ListOfPosn)
;; interp. 今生きているセルの座標だけを並べたリスト
;;         （死んでいるマスはリストに載せない）

(define WORLD0 empty)
(define WORLD-BLOCK
  (list (make-posn 1 1) (make-posn 1 2)
        (make-posn 2 1) (make-posn 2 2)))
```

**テンプレート**

まだ「何をする関数か」が決まっていないときの骨組みです。BSL では `...` をプレースホルダとして書けます（第2章参照）。

**署名をどう書くか（ステップ）**

1. 完成した関数には、理想的には「引数と返り値の型」を書きたい  
2. BSL では `(: 名前 シグネチャ)` という**コードとしての署名**が使える（例: `(: double (Number -> Number))`）  
3. ただし BSL のシグネチャでは、`Number` や `Boolean` などは使えるが、**`Posn` という型名は使えない**（未定義になる）  
4. そのため本線の `ListOfPosn` を扱う関数では、  
   - ノートや本文では `;; alive?: ListOfPosn Posn -> Boolean` のように**コメントで署名**する  
   - 付属の `.rkt` でも、Posn を含む署名はコメントにするか、緩い型（後述の `Any`）にする  
5. `Number` だけなら `(: …)` を積極的に使う（第4章の `survives?` など）  

```racket
;; この型を処理する関数の共通の骨
;; 署名（コメント）: ListOfPosn -> Any
(define (world-temp w)
  (cond
    [(empty? w) (...)]
    [else
     (... (first w)
          (world-temp (rest w)))]))
```

**所属判定と `alive?`**

- **所属判定** … 「この値は、このリストの中に**あるか**？」を true / false で答えること  
- **`alive?`** … ライフ用語での所属判定。第1引数が生存リスト、第2引数が調べたい座標。リストにあれば「今生きている」  

第1–2章の `alive?` と同じです。

```racket
;; alive?: ListOfPosn Posn -> Boolean
;; cells の中に cell があれば true（生きている）
(define (alive? cells cell)
  (cond
    [(empty? cells) false]
    [(equal? (first cells) cell) true]
    [else (alive? (rest cells) cell)]))

(check-expect (alive? WORLD-BLOCK (make-posn 1 1)) true)
(check-expect (alive? WORLD-BLOCK (make-posn 0 0)) false)
```

#### 3.3 データ定義 B: 密グリッド（理解用）

ここでは **A（スパース = 生存だけ）** と対比して、**B（密 = 全部のマス）** を書きます。

- A では 2×2 のブロックは生存4点だけ  
- B では同じ図を、例えば大きな枠の中の 0/1 の表として全部書く  

「比較する意味」は次のとおりです。

- 同じ盤を2通りで表せると、第4章の「生存リスト」が何を省略しているか分かる  
- 小さい例では密の方が目で追いやすい。大きい盤ではスパースの方が軽い  

```racket
;; CellValue is 0 or 1
;; interp. 0 = 死, 1 = 生

;; Row is ListOf CellValue   （1行ぶんの 0/1）
;; Grid is ListOf Row        （行を上から下へ並べた表）
;; interp. 上から下へ行、左から右へ列

(define TINY-GRID
  (list (list 0 1 0)
        (list 0 0 1)
        (list 1 1 1)))
```

**行番号 `r` と列番号 `c`（ステップ）**

1. 密グリッドは「表」だと考える  
2. **行番号 `r`** … 上から何行目か。いちばん上を `0` とする（プログラミングでよくある数え方）  
3. **列番号 `c`** … 左から何列目か。いちばん左を `0` とする  
4. 例: `TINY-GRID` の  
   - `r=0, c=1` → 上の行の真ん中 → `1`（生）  
   - `r=0, c=0` → 左上 → `0`（死）  
   - `r=2, c=0` → 下の行の左 → `1`（生）  
5. 関数 `grid-ref` は「表 `g` の、行 `r`、列 `c` のマスの値」を返す  

BSL では `list-ref` で「リストの n 番目」が取れます。行を取り、その行の列を取る、と二段階です。

デザインレシピでは、完成形の前に **スタブ**（中身はダミーの仮定義）を置く段階があります。  
スタブは「テストを先に動かすための実行可能なハリボテ」で、テンプレート変数 `...`（設計図の空欄）とは別物です（第2章・[Template Variables](https://docs.racket-lang.org/htdp-langs/beginner.html#%28part._beginner._.Template._.Variables%29)）。

次のスタブは**写経用の完成コードではありません**。ノートや途中ファイルで一時的に使う例です（`;` でコメントにしてあります）。

```racket
;; --- スタブ（一時的・完成時は消すか上書きする）---
;; grid-ref: Grid Number Number -> Number
;; (define (grid-ref g r c) 0)
```

完成形（付属コードと同じ。こちらを Run する）:

```racket
;; grid-ref: Grid Number Number -> Number
;; (grid-ref g r c) は g の r 行 c 列の 0 または 1
(: grid-ref (Any Number Number -> Number))
(define (grid-ref g r c)
  (list-ref (list-ref g r) c))

(check-expect (grid-ref TINY-GRID 0 1) 1)
(check-expect (grid-ref TINY-GRID 2 0) 1)
(check-expect (grid-ref TINY-GRID 0 0) 0)
```

**`Any` とは何か（ステップ）**

1. `(: grid-ref (…… -> Number))` の括弧の中は、「引数の型 → 返り値の型」  
2. **`Any`** … 「どんな値でもよい」という、いちばん緩い型の書き方  
3. ここでは第1引数 `g` を `Any` としている → シグネチャ上は「盤以外も受け取れる」と宣言している  
4. 実際の本体は `list-ref` なので、**本当はリストを渡す必要がある**。約束は緩く、中身でリストとして使う  

**`List` に厳密化する、とは何か（ステップ）**

1. 第1引数を `List` と書くと、「リスト以外を渡すな」と宣言できる  
2. 誤って数を渡したとき、BSL の検査が気づいてくれる可能性が上がる  
3. ただし本章の主目標は「デザインレシピの流れ」であり、シグネチャの細かい型名の議論ではない  
4. 初学者が `Any` と `List` のどちらが正しいかに長く止まると、認知負荷が上がりやすい  
5. よって本章では **`Any` のまま**にし、型の厳密さは後の章や発展課題へ回す  

#### 3.4 密グリッドと `ListOfPosn` の行き来（考え方）

言葉の復習:

- **密グリッド** … すべてのマスを 0/1 で書く表（3.3 の B）  
- **スパース（ListOfPosn）** … 生きている座標だけを並べるリスト（3.2 の A）  

同じ図形を2通りで表せると、第4章の「生存リスト」が何を意味するかがはっきりします。

- **密 → スパース**: 各マスを見て、値が 1 ならその `(列, 行)` をリストに `cons` する  
- **スパース → 密**: 先に全部 0 の表を用意し、リストにある座標だけ 1 にする  

**相互変換** … 上の「密→スパース」「スパース→密」のように、**同じ盤を別のデータ形に直すこと**です。変換しても「どこが生きているか」という事実は同じであるべきです。

付属コード `code/ch03-grid.rkt` では、小さな例で `grid-ref` とブロックの `ListOfPosn` を `check-expect` します。完全な相互変換関数は発展課題でも構いません。

#### 3.5 不変データ（世代を進める準備）

**「書き換える」と「新しく作る」**

1. 悪いイメージ: 今のリストの中身をその場で消したり書き換えたりする（破壊的更新）  
2. ライフの学習では、**入力の盤は残したまま、次の盤を別の値として返す**方がよい  
3. 理由:  
   - 1世代前と1世代後を同時に手元に置ける  
   - `check-expect` で「この入力 → この出力」と書きやすい  
   - バグのとき「どこを壊したか」が追いやすい  
4. 第2章の `cons` も「新しいリストを作る」ので、同じ発想  

ライフでは「今の盤」をその場で書き換えるより、**次の盤を新しく作って返す**方が、上記の意味で安全です。

- 入力の `ListOfPosn` は残したまま  
- `next-generation`（第4章）は新しいリストを返す  
- だからテストで「1世代前」と「1世代後」を同時に比較できる  

#### 3.6 まとめと第4章への橋

**この章で出てきた主な名前と役割**

- `WORLD0` / `WORLD-BLOCK` — 盤の例（生存リスト）  
- `world-temp` — ListOfPosn を処理する関数のテンプレート（骨）  
- `alive?` — その座標が生存リストにいるか（所属判定）  
- `TINY-GRID` — 密グリッドの小さな例  
- `grid-ref` — 密グリッドの行 `r`・列 `c` の 0/1 を取る  

- 本線のデータは **`ListOfPosn`（生存だけ・スパース）**  
- 密グリッドは説明用・小さな例用  
- データ定義 → テンプレート（`...`）→ スタブ → `check-expect` → 本体、の順  
- 次章で「近傍を数える」「次世代を返す」を、このデータに対して書く  

付属コードの実行は、**リポジトリのルート**（`mypublish-gameoflife/` や worktree のルート。`books/` の中ではない）から行ってください。

```bash
# カレントディレクトリがリポジトリ root であること
racket code/ch03-grid.rkt
```
