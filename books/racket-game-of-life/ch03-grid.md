#### 3.1 なぜ「グリッド」をデータで決めるか

画面の升目そのものをいきなり描く前に、**プログラムが扱う値の形**を決めます。

第2章のデザインレシピと同じように、次の**同じ順番**で進めます。

1. **データ定義** — 「どんな形の値を扱うか」をコメントで決める
2. **解釈（interp.）** — それが盤の何を意味するかを日本語で書く
3. **具体例** — `define` で実際の値を2〜3個置く
4. **テンプレート** — その形を処理する関数の骨（`cond` など）を `...` で書く
5. **スタブ** — 名前と引数だけ揃えた仮の関数（中身はダミー）
6. **`check-expect`** — 「この入力ならこの出力」を先に書く
7. **本体** — `...` やダミーを本物の式に置き換える

第1–2章では数やリストの長さなどでこの流れを練習しました。本章では同じ流れを、**ライフの盤面**に当てはめます。

- 形が決まれば、関数の `cond` の枝や使う部品（`first` / `rest` など）もほぼ決まる
- ライフでは「どこが生きているか」が中心なので、その表し方を選ぶ

本章では表し方を **2通り**紹介します。

- **本線（A）** … この本の第4章以降で**ずっと使う**やり方。学習のメインルート
- **理解用（B）** … 説明や小さな例のために使う補助。本線ではない

**A. 生存セルだけを並べる（スパース）**

- データ: `ListOfPosn`（第1–2章と同じ）
- 生きている座標だけをリストに入れる。死は「リストに無い」で表す
- 空の盤面は **`empty`**（下で詳しく）
- **スパース** … 「すき間が多い」「全部は書かない」という意味。死マスを0で埋めない

**`empty` とは（やや詳しく）**

1. `empty` は BSL に最初からある**値**です。「要素が0個のリスト」を表します。
2. **定数**です。自分で `(define empty …)` する必要はありません（組み込みの名前）。
3. **関数ではありません**。`(empty)` のように括弧を付けて呼ぶものではなく、そのまま `empty` と書きます。
4. 「リストが空か？」を調べるときは関数 **`empty?`** を使います。例: `(empty? empty)` は `true`。
5. 空の盤（誰も生きていない）は、スパース表現ではちょうど `empty` です。

**スパースを目で見る（ASCII）**

次は 5×5 の枠のイメージです。`#` が生、`.` が死。スパースでは **`#` の座標だけ**をリストに入れ、`.` は書きません。

```text
  列→  0 1 2 3 4
行 0   . . . . .
   1   . # # . .     ← (1,1) と (2,1) が生
   2   . # # . .     ← (1,2) と (2,2) が生（2×2 ブロック）
   3   . . . . .
   4   . . . . .

スパースのリスト（生きている4点だけ）:
  (list (make-posn 1 1) (make-posn 2 1)
        (make-posn 1 2) (make-posn 2 2))
※ 死のマスはリストに出てきません
```

**B. 升目を全部持つ（密グリッド）**

- データ: 行のリスト。各行は 0/1 のリスト
- すべてのマスを書くので、小さな図をそのままコードにしやすい
- **密** … 「すき間なく全部書く」という意味。3×3 なら9個の 0/1 が必ずある

同じ 2×2 ブロックを、密グリッド（0=死, 1=生）で書くと次のように**すべてのマス**が見えます。

```text
  列→  0 1 2 3 4
行 0   0 0 0 0 0
   1   0 1 1 0 0
   2   0 1 1 0 0
   3   0 0 0 0 0
   4   0 0 0 0 0

コードでは例えば（枠を小さくした 3×3 の別例）:
  (list (list 0 1 0)
        (list 0 0 1)
        (list 1 1 1))
↑ 9個すべてを書く = 密
```

**「正しさを `check-expect` で固定する」とは（ステップ）**

1. 関数を書いたあと、「合っているか」を目で見るだけだと、あとから壊しても気づきにくい
2. `check-expect` は「この入力なら、この出力であるべき」と**仕様をコードに残す**仕組み（第1章・第2章）
3. Run すると、仕様どおりなら成功、違えば失敗と分かる
4. だから「正しさの固定」＝ 期待する入出力をテストとして書いておく、という意味
5. 詳しい書き方とデザインレシピでの位置づけは **第1章 1.5・1.9**、リストでの使い方は **第2章** を参照

#### 3.2 データ定義 A: `ListOfPosn`（本線）

ここでは第2章で学んだ**リスト**を、盤面用に使います。復習をコードで示します。

**第2章のリスト（数の例・5つ）**

```racket
;; 1) 空リスト（定数 empty）
(define LON0 empty)

;; 2) 要素が1つ: 先頭 99、残り empty
(define LON1 (cons 99 empty))

;; 3) 要素が2つ: 4 の次が (6)
(define LON2 (cons 4 (cons 6 empty)))

;; 4) list は cons を重ねた書き方（読みやすい）
(define LON3 (list 1 8 5))
;; だいたい (cons 1 (cons 8 (cons 5 empty))) と同じ

;; 5) first / rest / empty? の使い方
;; (first LON2)  => 4
;; (rest LON2)   => (cons 6 empty)  つまり (list 6)
;; (empty? LON0) => true
;; (empty? LON1) => false
```

ポイント（第2章と同じ）:

- リストは `empty` か `(cons 先頭 残り)`
- 残りも同じ型のリスト（自己参照）
- 処理は `empty?` / `first` / `rest` のテンプレート

第2章では要素が「数」でした。本章の本線では要素が **`posn`（座標）** です。

**`posn` は組み込みの構造体**です。第1章の `define-struct cell ...` のように自分で型を宣言しなくても、BSL が最初から用意しています。

```racket
;; --- posn のイメージ（組み込み。自分で define-struct しない）---
;; だいたい次のような構造体だと思ってよい:
;;   (define-struct posn (x y))
;; 作る: (make-posn 横 縦)
;; 取る: (posn-x p)  (posn-y p)
;; 詳しくは第1章 1.6（構造体と posn）

;; ListOfPosn is one of:
;;  - empty
;;  - (cons Posn ListOfPosn)
;; interp. 今生きているセルの座標だけを並べたリスト
;;         （死んでいるマスはリストに載せない）

(define WORLD0 empty)   ; 誰もいない盤 = empty という定数値

;; 2×2 ブロック（ASCII で見たとおり4点）
(define WORLD-BLOCK
  (list (make-posn 1 1)   ; x=1, y=1
        (make-posn 1 2)   ; x=1, y=2
        (make-posn 2 1)
        (make-posn 2 2)))
```

**テンプレート**

まだ「何をする関数か」が決まっていないときの骨組みです。BSL では `...` をプレースホルダとして書けます（第2章参照）。

**引数・返り値と署名（ステップ）**

1. **引数** … 関数に**渡す**値。`(alive? WORLD-BLOCK (make-posn 1 1))` なら、第1引数がリスト、第2引数が座標
2. **返り値** … 関数が**計算して返す**値。`alive?` なら `true` か `false`
3. **署名** … 「どんな引数を何個受け取り、どんな型を返すか」という約束
4. コメントで書く例: `;; alive?: ListOfPosn Posn -> Boolean`（左が引数、右が返り値）
5. コードで書く例: `(: double (Number -> Number))`（詳細は第2章「シグネチャ」節）

上記 1〜4 で例に出した **`alive?` の完成コードは、すぐ下の「所属判定と `alive?`」に出てきます**（先に署名とテンプレートの型を押さえ、それから本体を見る順です）。

**BSL の `(: …)` で Posn が使えない件（ステップ）**

1. 完成した関数には、理想的には「引数と返り値の型」を書きたい
2. BSL では `(: 名前 シグネチャ)` という**コードとしての署名**が使える（例: `(: double (Number -> Number))`）
3. ただし BSL のシグネチャでは、`Number` や `Boolean` などは使えるが、**`Posn` という型名は使えない**（未定義エラーになる）
4. そのため本線の `ListOfPosn` を扱う関数では、
   - ノートや本文では `;; alive?: ListOfPosn Posn -> Boolean` のように**コメントで署名**する
   - 付属の `.rkt` でも、Posn を含む署名はコメントにするか、緩い型（後述の `Any`）にする
5. `Number` だけなら `(: …)` を積極的に使う（第4章の `survives?` など）

**テンプレート `world-temp` と署名の対応**

コメント署名 `ListOfPosn -> Any` と、関数本体の各部品は次のように対応します。

- **`ListOfPosn`（引数の型）** → 仮引数 `w` … 生存リストそのもの
- **`Any`（返り値の型）** → `(...)` が埋まる場所 … まだ決めていない「この関数が返す何か」
- **`empty` の枝** → `[(empty? w) (...)]` … リストが空のときの答え（プレースホルダ）
- **`cons` の枝（先頭）** → `(first w)` … 先頭の `Posn`
- **`cons` の枝（残り）** → `(world-temp (rest w))` … 残りリストへの再帰
- **全体の組み合わせ** → `else` の外側の `(...)` … 先頭と再帰結果をどう足すか（未定）


```racket
;; この型を処理する関数の共通の骨
;; 署名（コメント）: ListOfPosn -> Any
;;   ListOfPosn  ↔  引数 w
;;   Any         ↔  各 (...) を埋めた結果（まだ決まっていない返り値）
(define (world-temp w)                 ; w : ListOfPosn
  (cond
    [(empty? w) (...)]                 ; 空のときの返り値（Any の候補）
    [else
     (... (first w)                    ; 先頭 Posn をどう使うか
          (world-temp (rest w)))]))    ; 残りへの再帰結果をどう使うか
```

`alive?` は、この骨に **第2引数 `cell` を足し**、空なら `false`、先頭が一致なら `true`、そうでなければ再帰、と `(...)` を埋めたものです（次節）。


**所属判定と `alive?`**

- **所属判定** … 「この値は、このリストの中に**あるか**？」を true / false で答えること
- **`alive?`** … ライフ用語での所属判定。第1引数が生存リスト、第2引数が調べたい座標。リストにあれば「今生きている」

第1–2章の `alive?` と同じです。上で予告した **署名の具体例が、ここで完成形として出てきます**。

```racket
;; alive?: ListOfPosn Posn -> Boolean
;; 引数 cells = 生存リスト, cell = 調べたい座標
;; 返り値 = リストにいれば true（生きている）
(define (alive? cells cell)
  (cond
    [(empty? cells) false]
    [(equal? (first cells) cell) true]
    [else (alive? (rest cells) cell)]))

(check-expect (alive? WORLD-BLOCK (make-posn 1 1)) true)
(check-expect (alive? WORLD-BLOCK (make-posn 0 0)) false)
```

#### 3.3 データ定義 B: 密グリッド（理解用）

ここでは **A（スパース = 生存だけ）** と対比して、**B（密 = 全部のマス）** を書きます。

- A では 2×2 のブロックは生存4点だけ
- B では同じ図を、例えば大きな枠の中の 0/1 の表として全部書く

「比較する意味」は次のとおりです。

- 同じ盤を2通りで表せると、第4章の「生存リスト」が何を省略しているか分かる
- 小さい例では密の方が目で追いやすい。大きい盤ではスパースの方が軽い

```racket
;; CellValue is 0 or 1
;; interp. 0 = 死, 1 = 生

;; Row is ListOf CellValue   （1行ぶんの 0/1）
;; Grid is ListOf Row        （行を上から下へ並べた表）
;; interp. 上から下へ行、左から右へ列

(define TINY-GRID
  (list (list 0 1 0)
        (list 0 0 1)
        (list 1 1 1)))
```

目視（`TINY-GRID`）:

```text
  列 0 1 2
行0  0 1 0
行1  0 0 1
行2  1 1 1
```

**行番号 `r` と列番号 `c`（ステップ）**

1. 密グリッドは「表」だと考える
2. **行番号 `r`** … 上から何行目か。いちばん上を `0` とする
3. **列番号 `c`** … 左から何列目か。いちばん左を `0` とする
4. 例: `TINY-GRID` の
   - `r=0, c=1` → 上の行の真ん中 → `1`（生）
   - `r=0, c=0` → 左上 → `0`（死）
   - `r=2, c=0` → 下の行の左 → `1`（生）
5. 関数 `grid-ref` は「表 `g` の、行 `r`、列 `c` のマスの値」を返す

BSL では `list-ref` で「リストの n 番目」が取れます。行を取り、その行の列を取る、と二段階です。

デザインレシピでは、完成形の前に **スタブ**（中身はダミーの仮定義）を置く段階があります。
スタブは「テストを先に動かすための実行可能なハリボテ」で、テンプレート変数 `...`（設計図の空欄）とは別物です（第2章・[Template Variables](https://docs.racket-lang.org/htdp-langs/beginner.html#%28part._beginner._.Template._.Variables%29)）。

次のスタブは**写経用の完成コードではありません**。ノートや途中ファイルで一時的に使う例です（`;` でコメントにしてあります）。

```racket
;; --- スタブ（一時的・完成時は消すか上書きする）---
;; grid-ref: Grid Number Number -> Number
;; (define (grid-ref g r c) 0)
```

完成形（付属コードと同じ。こちらを Run する）:

**`list-ref` とは**

- **`(list-ref lst n)`** … リスト `lst` の **先頭を 0 番目** として、`n` 番目の要素を1つ返す組み込み関数
- 例: `(list-ref (list "a" "b" "c") 1)` → `"b"`
- 範囲外（`n` が長すぎる）だとエラーになる
- 公式: [Beginning Student — `list-ref`](https://docs.racket-lang.org/htdp-langs/beginner.html#%28def._htdp-beginner._%28%28lib._lang%2Fhtdp-beginner..rkt%29._list-ref%29%29)（[同ページ一覧](https://docs.racket-lang.org/htdp-langs/beginner.html) の Pre-defined Functions 付近）

`grid-ref` ではこれを **二段** 使います。

1. `(list-ref g r)` … 表 `g`（行のリスト）から **行 `r`** を取り出す → その行は 0/1 のリスト
2. `(list-ref … c)` … その行から **列 `c`** のマスを取り出す → 0 または 1

```racket
;; grid-ref: Grid Number Number -> Number
;; 引数: g = 表, r = 行番号, c = 列番号
;; 返り値: そのマスの 0 または 1
(: grid-ref (Any Number Number -> Number))
(define (grid-ref g r c)
  (list-ref (list-ref g r) c))   ; 内側で行、外側で列

(check-expect (grid-ref TINY-GRID 0 1) 1)
(check-expect (grid-ref TINY-GRID 2 0) 1)
(check-expect (grid-ref TINY-GRID 0 0) 0)
```

**`Any` とは何か（ステップ）**

1. `(: grid-ref (…… -> Number))` の括弧の中は、「引数の型 → 返り値の型」
2. 矢印の**左側**が引数の型、**右側**が返り値の型
3. **`Any`** … 「どんな値でもよい」という、いちばん緩い型の書き方
4. ここでは第1引数 `g` を `Any` としている → シグネチャ上は「盤以外も受け取れる」と**宣言している**
5. 実際の本体は `list-ref` なので、**本当はリストを渡す必要がある**。約束は緩く、中身でリストとして使う

**「宣言できる」とはどういう意味か（ステップ）**

1. 署名はコンピュータへの**約束**である
2. 「宣言する」＝「この関数はこういう型の引数を受け取り、こういう型を返す」とプログラムに書くこと
3. `Any` と宣言すると、「どんな型でも受け取る」と約束したことになる（検査が緩い）
4. `List` と宣言すると、「リスト以外は渡すな」と約束したことになる（検査が少し厳しくなる）
5. 約束に合わない使い方をすると、BSL はシグネチャ違反として教えてくれることがある
6. 宣言を厳しくするほど、誤用を早く見つけやすい一方、書き方の学習コストも上がる

**`List` に厳密化する、とは何か（ステップ）**

1. 署名（シグネチャー）で第1引数を `List` と書くと、「リスト以外を渡すな」と宣言できる
2. 誤って数を渡したとき、BSL の検査が気づいてくれる可能性が上がる
3. ただし本章の主目標は「デザインレシピの流れ」であり、シグネチャの細かい型名の議論ではない
4. 初学者が `Any` と `List` のどちらが正しいかに長く止まると、認知負荷が上がりやすい
5. よって本章では **`Any` のまま**にし、型の厳密さは後の章や発展課題へ回す

#### 3.4 密グリッドと `ListOfPosn` の行き来（考え方）

言葉の復習:

- **密グリッド** … すべてのマスを 0/1 で書く表（3.3 の B）。ASCII ではマスが全部見える
- **スパース（ListOfPosn）** … 生きている座標だけを並べるリスト（3.2 の A）。死はリストに無い

同じ図形を2通りで表せると、第4章の「生存リスト」が何を意味するかがはっきりします。

- **密 → スパース**: 各マスを見て、値が 1 ならその `(列, 行)` をリストに `cons` する
- **スパース → 密**: 先に全部 0 の表を用意し、リストにある座標だけ 1 にする

**相互変換** … 上の「密→スパース」「スパース→密」のように、**同じ盤を別のデータ形に直すこと**です。変換しても「どこが生きているか」という事実は同じであるべきです。

付属コード `code/ch03-grid.rkt` では、小さな例で `grid-ref` とブロックの `ListOfPosn` を `check-expect` します。完全な相互変換関数は発展課題でも構いません。

#### 3.5 不変データ（世代を進める準備）

**「書き換える」と「新しく作る」**

1. 悪いイメージ: 今のリストの中身をその場で消したり書き換えたりする（破壊的更新）
2. ライフの学習では、**入力の盤は残したまま、次の盤を別の値として返す**方がよい
3. 理由:
   - 1世代前と1世代後を同時に手元に置ける
   - `check-expect` で「この入力 → この出力」と書きやすい
   - バグのとき「どこを壊したか」が追いやすい
4. 第2章の `cons` も「新しいリストを作る」ので、同じ発想

ライフでは「今の盤」をその場で書き換えるより、**次の盤を新しく作って返す**方が、上記の意味で安全です。

- 入力の `ListOfPosn` は残したまま
- `next-generation`（第4章）は新しいリストを返す
- だからテストで「1世代前」と「1世代後」を同時に比較できる

#### 3.6 まとめと第4章への橋

**この章で出てきた主な名前と役割**

- `empty` — 空リストという**定数値**（空の盤）
- `WORLD0` / `WORLD-BLOCK` — 盤の例（生存リスト）
- `world-temp` — ListOfPosn を処理する関数のテンプレート（骨）
- `alive?` — その座標が生存リストにいるか（所属判定）
- `TINY-GRID` — 密グリッドの小さな例
- `grid-ref` — 密グリッドの行 `r`・列 `c` の 0/1 を取る

**本章の流れをステップで振り返る**

1. **本線はスパース（`ListOfPosn`）**
   生きている座標だけをリストに入れる。死は載せない。空の盤は `empty`。
2. **理解用に密グリッドもある**
   すべてのマスを 0/1 で書く。小さな例の目視に向く。
3. **進め方はデザインレシピの順**
   データ定義 → テンプレート（`...`）→ スタブ → `check-expect` → 本体。第1–2章と同じ型紙。
4. **次章への接続**
