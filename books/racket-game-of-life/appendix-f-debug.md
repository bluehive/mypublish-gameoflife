#### F.0 まず心がけること（Scratch 経験者向け）

Scratch でも、「予想と違う動き」のときは次をしました。

1. **どこまで正しいか**を切り分ける（最初のブロックから順に）  
2. 変数や見た目で**今の値**を確認する  
3. 小さな例で試す  

Racket（BSL）でも同じです。いきなり全部を疑わず、**小さな式**と **`check-expect`** から攻めます。

本編の付属コードは、多くの場合 **`check-expect` が仕様**です。まず Run して、どのテストが赤（失敗）かを見てください。

#### F.1 プリントデバッグ（途中の値を見る）

**何をするか** … 計算の途中で、「今この値は何か」を画面に出す。

Racket は「全部が純粋関数」というわけではないので、学習中は途中表示を差し込みやすいです。ただし **Beginning Student（BSL）では `printf` / `begin` が使えない**ことが多いです。

| 言語 | やりやすいこと |
| --- | --- |
| **BSL（本の本線）** | 相互作用ウィンドウで式を評価する。関数を小さく分け、`check-expect` で中間結果を固定する |
| **Intermediate 以降** | `(printf "x is currently ~v\n" x)` や `begin` で「表示してから値を返す」 |

Intermediate などで使える例（参考・本線 BSL では動かないことがある）:

```racket
;; 参考: Intermediate など
;; (begin (printf "x is currently ~v\n" x) x)
```

BSL での現実的な代わり:

1. 疑わしい式だけを相互作用ウィンドウに貼って評価する  
2. 小さな関数に分け、それぞれに `check-expect` を付ける  
3. 第2章の**トレース**（式を紙に展開する）で、手計算と Run の結果を比べる  

参考: [debugging-in-racket（プリントの例）](https://www.brinckerhoff.org/clements/2252-csc430/Files/debugging-in-racket.html)

#### F.3 REPL（相互作用ウィンドウ）でのデバッグ

**REPL** とは、1行ずつ式を入れて、すぐ結果が返ってくる対話画面です。DrRacket では下側の**相互作用ウィンドウ**がそれに近い役割です。

やるとよいこと:

1. 定義ウィンドウを Run して、関数を読み込む  
2. 相互作用側で `(my-length (cons 4 (cons 6 empty)))` のように**小さな例**を試す  
3. エラーメッセージを最後から読む（「どの関数の、何が不満か」）  
4. スタックトレースだけでなく、「引数を変えたらどうなるか」を試す  

Scratch の「緑の旗で全部動かす」より、**部品を単体で叩く**イメージです。

参考（考え方）: [Common Lisp のデバッグ料理本（対話の発想）](https://lispcookbook.github.io/cl-cookbook/debugging.html)  
※言語は違いますが、「対話で探る」発想の参考です。

#### F.4 トレース（呼び出しの記録）— 考え方

**trace** とは、「この関数がどんな引数で呼ばれ、何を返したか」を記録して見せる考え方です。再帰では、第2章の**計算トレース**（式の展開）が、まさにその紙バージョンです。

BSL の本線では、次を推奨します。

1. 紙またはメモに、`(my-length …) = … = …` と展開する（第2章 2.3）  
2. `check-expect` の期待値と、手計算の最終結果を一致させる  
3. 言語を上げたあとで、処理系の trace 機能やステップ実行を使う（発展）  

「ブレークポイントで止める」は IDE の話です。まずは **小さな入力・手計算・自動テスト** の三点セットで十分です。
