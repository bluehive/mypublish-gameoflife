#### F.0 まず心がけること（Scratch 経験者向け）

Scratch でも、「予想と違う動き」のときは次をしました。

1. **どこまで正しいか**を切り分ける（最初のブロックから順に）
2. 変数や見た目で**今の値**を確認する
3. 小さな例で試す

Racket（BSL）でもと同じです。いきなり全部を疑わず、**小さな式**と **`check-expect`** から攻めます。

本編の付属コードは、多くの場合 **`check-expect` が仕様**です。まず Run して、どのテストが赤（失敗）かを見てください。

#### F.1 プリントデバッグ（途中の値を見る）

**何をするか** … 計算の途中で、「今この値は何か」を画面に出す。

Racket は「全部が純粋関数」というわけではないので、学習中は途中表示を差し込みやすいです。ただし **Beginning Student（BSL）では `printf` / `begin` が使えません**。これには以下のような理由と代わりの方法があります。

* **なぜ BSL では `printf` や画面表示が使えないのか？**
  * Scratchの緑色の「演算ブロック（足し算や文字列の合体など）」の中に、吹き出しを出す「言う」ブロックをハメ込めないのと同じです。BSLの式は純粋な「計算機」なので、計算の途中で「画面に文字を出す」という別のアクションを割り込ませる余地がありません。
* **BSL（本の本線）での現実的な解決策**:
  * 相互作用ウィンドウ（REPL）で式を直接評価して確認する。
  * 関数を小さく分け、それぞれの機能ごとに `check-expect` で中間結果をテスト・固定する。
  * 第2章の**トレース**（式を紙に展開する）で、手計算と Run の結果を比べる。
* **Intermediate Student Language 以降（発展）**:
  * `(printf "x is currently ~v\n" x)` や `begin` を使って、「画面に値を表示してから、本来の計算結果を返す」というプリントデバッグが可能です。

Intermediate などで使える例（参考・本線 BSL では動かないことがあります）:

```racket
;; 参考: Intermediate など
;; (begin (printf "x is currently ~v\n" x) x)
```

参考: [debugging-in-racket（プリントの例）](https://www.brinckerhoff.org/clements/2252-csc430/Files/debugging-in-racket.html)

#### F.3 REPL（相互作用ウィンドウ）でのデバッグ

**REPL** とは、1行ずつ式を入れて、すぐ結果が返ってくる対話画面です。DrRacket では下側の**相互作用ウィンドウ**がそれに近い役割です。

やるとよいこと:

1. 定義ウィンドウを Run して、関数を読み込む
2. 相互作用側で `(my-length (cons 4 (cons 6 empty)))` のように**小さな例**を試す
3. エラーメッセージを最後から読む（「どの関数の、何が不満か」）
4. スタックトレースだけでなく、「引数を変えたらどうなるか」を試す

Scratch の「緑の旗で全部動かす」より、**部品を単体で叩く**イメージです。

#### F.4 トレース（呼び出しの記録）— 考え方

**trace（トレース）** とは、「この関数がどんな引数で呼ばれ、何を返したか」を記録して見せる考え方です。

再帰関数が思ったとおりに動かないとき、BSLの本線では**「紙やメモを使って、手動で計算をトレースする（式を展開する）こと」**を最も推奨します。

1. **紙やメモに、`(my-length …) = … = …` と手動で展開する（最も推奨）**  
   第2章で学んだように、式を自分の手で1行ずつ置き換えていくことで、プログラムが「どこで止まるべきか（基本ケース）」「どのように縮小しているか（再帰ケース）」を頭の中で論理的に整理できます。
2. **`check-expect` の期待値と、手計算の最終結果を一致させる**  
   手計算した結果と、プログラム上のテスト期待値がずれている場合、そもそも「問題の解き方（アルゴリズム）」が間違っていると分かります。
3. **言語レベルを上げたあとで、処理系の `trace` 機能を使う（発展）**  

「ブレークポイントで一時停止する」といった複雑なデバッガの使い方は、この段階では不要です。まずは **「手動のトレース・手計算・自動テスト（check-expect）」** の3点セットを使いこなすことが、バグを減らす一番の近道です。

> **【コラム】DrRacket の「Stepper」機能による可視化**
> 
> ※この「Stepper（ステッパー）」機能は、DrRacket（GUIエディタ）限定の補助ツールであり、本編で学ぶデバッグの必須手順（本線）としては扱いません。ただし、「どうしても再帰の計算が頭の中でイメージできない」という場合の強力な救済手段となります。
> 
> **DrRacket 「Stepper」機能：ステップバイステップ操作ガイド**
> 
> 1. **テスト用のコードを書く**  
>    DrRacketの定義ウィンドウ（上の段）に、観察したい関数と、具体的な引数を与えた呼び出し式を記述します。
>    ```racket
>    (define (my-length lon)
>      (cond
>        [(empty? lon) 0]
>        [else (+ 1 (my-length (rest lon)))]))
> 
>    (my-length (list 4 6))  ; ← これを観察対象にします
>    ```
> 2. **Stepper（ステップ実行）を起動する**  
>    DrRacket画面の右上ツールバーにある **「Step」ボタン** をクリックします。  
>    （キーボードショートカット： `Ctrl+Shift+X` (Windows/Linux) / `Cmd+Shift+X` (Mac) ）
> 3. **専用ウィンドウの確認**  
>    別ウィンドウで「Stepper」が起動します。画面は左右に分割されており、左側が「評価前の式（Before）」、右側が「評価後の式（After）」を示します。
> 4. **ハイライト（評価箇所）の確認**  
>    コンピュータが「今まさに計算しようとしている部分」が **緑色にハイライト** されて左側に表示されます。
> 5. **ステップを進める・戻す**  
>    * **「Next」ボタン**（または `→` キー）を押すと、緑色のハイライト部分が計算（評価）され、右側の形に置き換わって次のステップへ進みます。
>    * **「Previous」ボタン**（または `←` キー）を押すと、1ステップ前の状態に時間を巻き戻せます。
> 6. **再帰展開 of ビジュアル化**  
>    「Next」を押し続けると、再帰関数がどのように展開していくかを目視できます。
>    * `(my-length (list 4 6))`
>    * `(+ 1 (my-length (list 6)))`
>    * `(+ 1 (+ 1 (my-length empty)))`
>    * `(my-length empty)` が基本ケースに到達し、`0` に置き換わる。
>    * 内側の足し算が順次解決されていく（`(+ 1 (+ 1 0))` → `(+ 1 1)` → `2`）。
> 7. **計算完了**  
>    最終値である `2` に到達するとステップ実行が終了します。バグがある場合は、この過程の中で「どの瞬間（ステップ）で間違った条件分岐（`cond`）に流れたか」をピンポイントで確認できます。
