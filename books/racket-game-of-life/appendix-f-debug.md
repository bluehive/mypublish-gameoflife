#### F.0 まず心がけること（Scratch 経験者向け）

Scratch でも、「予想と違う動き」のときは次をしました。

1. **どこまで正しいか**を切り分ける（最初のブロックから順に）  
2. 変数や見た目で**今の値**を確認する  
3. 小さな例で試す  

Racket（BSL）でも同じです。いきなり全部を疑わず、**小さな式**と **`check-expect`** から攻めます。

本編の付属コードは、多くの場合 **`check-expect` が仕様**です。まず Run して、どのテストが赤（失敗）かを見てください。

エラーが出たときは、赤い文字だけ見て止まらず、**F.4 のスタックトレース**で「どの関数の、どの呼び出しの途中か」を追います。

#### F.1 プリントデバッグ（途中の値を見る）

**何をするか** … 計算の途中で、「今この値は何か」を画面に出す。

Racket は「全部が純粋関数」というわけではないので、学習中は途中表示を差し込みやすいです。ただし **Beginning Student（BSL）では `printf` / `begin` が使えない**ことが多いです。

| 言語 | やりやすいこと |
| --- | --- |
| **BSL（本の本線）** | 相互作用ウィンドウで式を評価する。関数を小さく分け、`check-expect` で中間結果を固定する |
| **Intermediate 以降** | `(printf "x is currently ~v\n" x)` や `begin` で「表示してから値を返す」 |

Intermediate などで使える例（参考・本線 BSL では動かないことがある）:

```racket
;; 参考: Intermediate など
;; (begin (printf "x is currently ~v\n" x) x)
```

BSL での現実的な代わり:

1. 疑わしい式だけを相互作用ウィンドウに貼って評価する  
2. 小さな関数に分け、それぞれに `check-expect` を付ける  
3. **エラー時は F.4 のスタックトレース**で「どこで止まったか」を見る  

参考: [debugging-in-racket（プリントの例）](https://www.brinckerhoff.org/clements/2252-csc430/Files/debugging-in-racket.html)

#### F.3 REPL（相互作用ウィンドウ）でのデバッグ

**REPL** とは、1行ずつ式を入れて、すぐ結果が返ってくる対話画面です。DrRacket では下側の**相互作用ウィンドウ**がそれに近い役割です。

やるとよいこと:

1. 定義ウィンドウを Run して、関数を読み込む  
2. 相互作用側で `(my-length (cons 4 (cons 6 empty)))` のように**小さな例**を試す  
3. エラーメッセージを読む（「どの関数の、何が不満か」）  
4. 赤いバグ印や「詳細」から **スタックトレース**を開く（**F.4**）  
5. 引数を変えて、「どの入力で壊れるか」を試す  

Scratch の「緑の旗で全部動かす」より、**部品を単体で叩く**イメージです。

参考（考え方）: [Common Lisp のデバッグ料理本（対話の発想）](https://lispcookbook.github.io/cl-cookbook/debugging.html)  
※言語は違いますが、「対話で探る」発想の参考です。

#### F.4 DrRacket のスタックトレース（本節のメイン）

ここがこの付録の中心です。  
「紙に式を書いて展開する」のではなく、**DrRacket がエラー時に見せてくれる呼び出し履歴（スタックトレース）**の使い方を、ステップで説明します。

##### F.4.1 スタックとは何か（ASCII でイメージ）

関数が別の関数を呼ぶとき、コンピュータは「今やっている仕事」を**積み重ね**て覚えます。この積み重ねを **呼び出しスタック（call stack）** と呼びます。

イメージは、机の上にメモ用紙を重ねる感じです。

- **新しい関数を呼ぶ** … 新しいメモを**一番上に置く**（push）  
- **その関数が終わる** … 一番上のメモを**捨てて下の仕事に戻る**（pop）  
- **エラーが起きる** … いま一番上にあるメモが「事故現場」。その下のメモが「誰から呼ばれてここに来たか」

```text
        ↑ 新しい呼び出し（上が「いま実行中」）
   +------------------+
   |  my-length       |  ← いまここでエラー、など
   |  引数: empty 付近 |
   +------------------+
   |  my-length       |  ← ひとつ外側の呼び出し
   |  引数: (cons 6 …) |
   +------------------+
   |  my-length       |  ← もっと外側
   |  引数: (cons 4 …) |
   +------------------+
   |  (相互作用の式)   |  ← いちばん最初に評価した式
   +------------------+
        ↓ 古い呼び出し（下が「入り口に近い」）
```

**スタックトレース（stack trace / backtrace）** とは、エラーが起きた瞬間のこの積み重ねを、DrRacket が表示したものです。

- **いちばん上（または一覧の先頭付近）** … エラーが検出された場所に近い  
- **下の方** … そこに至るまでの呼び出し経路（誰が誰を呼んだか）

Scratch に近い言い方をすると:

- スタック … 「いま動いているスクリプトが、どのブロックの途中か」の履歴  
- トレース … その履歴のプリントアウト  

##### F.4.2 エラーと「普通の赤テスト」のちがい

| 現象 | 意味 | スタックトレース |
| --- | --- | --- |
| `check-expect` が赤 | 計算は最後まで終わったが、**答えが期待と違う** | 出ないことが多い（失敗は「不一致」） |
| 実行時エラー（赤いメッセージ） | 途中で**止まった**（定義なし、契約違反、空リストに `first` など） | **ここでスタックが役立つ** |
| シグネチャ違反 | 型の約束と違う使い方 | テスト結果と一緒に出ることがある |

つまり F.4 は主に、**「落ちた」「止まった」とき**に使います。  
「答えが 2 なのに 3 になった」だけのときは、F.3 で小さな例を試す＋第2章の式の展開（理解用）の方が向きます。

##### F.4.3 事前準備（ステップ）

1. **DrRacket を起動**する  
2. 対象のファイルを開く（例: `code/ch02-recursion.rkt`）  
3. 言語が本文と一致しているか確認する  
   - ファイル先頭が `#lang htdp/bsl` なら Beginning Student 系  
   - メニュー **Language（言語）→ Choose Language…** と食い違わないようにする  
4. **デバッグ情報を有効にする**（スタックが出やすい）  
   - **Language → Choose Language… → Show Details（詳細を表示）**  
   - **Dynamic Properties / Debugging** あたりで、デバッグを切っていないことを確認  
   - （版によって文言が少し違います。「No debugging / デバッグなし」になっていたら、通常のデバッグに戻す）  
5. 定義ウィンドウを **Run（実行）** する  

デバッグを切っているとプログラムは速いことがありますが、**エラー時のスタックが薄くなる**ことがあります。学習中はデバッグ ON を推奨します。

参考: [DrRacket のデバッグ関連](https://docs.racket-lang.org/tools/drracket_debug.html) / [Racket のデバッグ概要](https://docs.racket-lang.org/reference/debugging.html)

##### F.4.4 わざと小さなエラーを出して、スタックを見る（練習手順）

安全な練習用に、**短いファイル**や相互作用だけで試せます。  
（付属の完成コードを壊したくなければ、新規ファイルに書いてください。）

**例A: 空リストに `first`（よくある事故）**

定義ウィンドウ:

```racket
#lang htdp/bsl

(define (head lon)
  (first lon))
```

手順:

1. Run する  
2. 相互作用ウィンドウで次を入力して Enter  

```racket
(head empty)
```

3. 赤いエラーメッセージが出る（「空リストに first」など）  
4. メッセージの近くにある **バグ印・赤い×・リンク・「Show」系** をクリックする  
5. **バックトレース / スタック**のウィンドウまたは展開表示を開く  

（DrRacket の版によって、エラー行の左のアイコンをクリック、またはメッセージ内のリンク、という違いがあります。ポイントは **「詳細な呼び出し履歴を開く」** です。）

**例B: 再帰で呼び出しが重なる**

```racket
#lang htdp/bsl

(define (my-length lon)
  (cond
    [(empty? lon) 0]
    [else (+ 1 (my-length (rest lon)))]))

(define (boom lon)
  (my-length (rest lon)))  ; lon が短すぎると rest で事故ることがある
```

相互作用:

```racket
(boom empty)
```

`rest` が空で失敗すると、スタックにはだいたい次のような**経路**が見えます（文言は環境で異なる）。

```text
エラー: rest: expects a non-empty list  …など
  呼ばれた経路のイメージ:

  (boom empty)          ← あなたが入力した式
    → (my-length …)     ← boom の中
    → (rest lon)        ← 事故現場に近い
```

ここで読み取ることは一つです。

> **「どの関数の、どの呼び出しの途中で止まったか」**

##### F.4.5 スタックトレースの読み方（ステップ）

表示を上から（または「内側から」）読む手順です。

1. **エラー本文を1行で言い換える**  
   例: 「空のリストに first した」「数が必要なのにリストが来た」  
2. **いちばん内側（事故現場）**を探す  
   リストの先頭付近やハイライトされた行番号  
3. **ひとつ外側**を見る  
   「誰がその関数を呼んだか」  
4. **入り口**まで戻る  
   相互作用に打った式、または `check-expect` の左辺、Run 時に評価された式  
5. **自分のコードの行**に飛ぶ  
   スタックの項目をクリックすると、定義ウィンドウの該当行が強調されることが多い  
6. **引数の値**が分かればメモする  
   「empty のときに first した」など、再現条件を一文にする  
7. **直す**  
   基本ケースを足す、`empty?` を先に書く、呼び出し側の引数を直す、など  
8. **同じ操作をもう一度**  
   直ったか、スタックが出なくなったかを確認する  

##### F.4.6 再帰のとき、スタックはどう積まれるか

第2章の `my-length` で、正常な計算中のスタックはだいたい次のように**一時的に**厚くなります。

```text
評価: (my-length (cons 4 (cons 6 empty)))

ステップイメージ（呼び出しが深くなる）:

  [1] my-length( 4,6 のリスト )
        └─ 中で my-length( 6 のリスト ) を待つ
              └─ 中で my-length( empty ) を待つ
                    └─ 0 を返す   ← 基本ケース

戻るとき（答えを持ち帰る）:

  [1] (+ 1 0) → 1
  [0] (+ 1 1) → 2
```

ASCII でもう一度（積み上がってから崩れる）:

```text
深さ3:  my-length(empty)     → 0 を返して消える
深さ2:  my-length(6…)        → (+ 1 0) を計算して消える
深さ1:  my-length(4,6…)      → (+ 1 1) を計算して消える
深さ0:  相互作用の式         → 結果 2
```

**エラーのとき**は、途中の深さで止まったまま履歴が残ります。  
だからスタックを見ると、「何回目の再帰で、どんな引数のときに落ちたか」が分かります。

Scratch の「〜まで繰り返す」との対応（第2章）:

- 基本ケースに入る前に落ちる … 止まる条件が足りない／先に `first` や `rest` している  
- 何度も同じ形で積まれる … リストが短くなっていない（無限再帰に近い）  

##### F.4.7 よくあるパターンと、スタックのヒント

| 症状の例 | スタックで見えがちなこと | まず疑うこと |
| --- | --- | --- |
| `first` / `rest` のエラー | 事故現場が `first` や `rest`、その外側が自分の関数 | `empty?` を先に書いていない |
| 名前が無い | 未定義の関数名 | つづり、Run 忘れ、別ファイルの関数 |
| 引数の個数違い | 呼び出し行がハイライト | `(f a b)` なのに定義は1引数、など |
| 再帰が終わらない／深くなりすぎ | 同じ関数名が何度も並ぶ | 基本ケースが無い、`rest` していない |
| `check-expect` だけ赤 | スタックより「実際の値 / 期待値」 | ロジックの間違い（F.3 で小さな例） |

##### F.4.8 Stepper（ステップ実行）とのちがい

DrRacket の教学言語には **Stepper（ステップ実行）** もあります（ツールバーの「Step」など。版で位置が違う）。

| 道具 | 何が分かるか | 向いている場面 |
| --- | --- | --- |
| **スタックトレース** | エラー時、「誰が誰を呼んでいたか」 | 落ちた・止まった |
| **Stepper** | 式が代数のように1手ずつ書き換わる様子 | 正常な計算の理解（第2章の展開に近い） |

この F.4 の本題は **スタックトレース**です。  
Stepper は「紙の式展開をソフトがやってくれる」理解用で、エラー調査の主役はスタックです。

##### F.4.9 短い練習メニュー（5分）

1. 新規 BSL ファイルで `(first empty)` を相互作用に打ち、エラーを出す  
2. バグ印／詳細からスタック（またはバックトレース）を開く  
3. 表示された項目をクリックし、どの行が光るか見る  
4. `(define (f x) (first x))` と `(f empty)` に変え、**自分の関数名が経路に出るか**を確認する  
5. `(define (f x) (cond [(empty? x) 0] [else (first x)]))` に直し、`(f empty)` がエラーにならないことを確認する  

これで、「スタックは怖いログ」ではなく、**迷子になったときの地図**になります。

#### F.6 困ったときの短いチェックリスト

1. `#lang` と DrRacket の言語レベルは一致しているか  
2. デバッグ（スタックが出る設定）を切っていないか（F.4.3）  
3. 失敗しているのは **エラー**か **`check-expect` の不一致**か  
4. エラーならスタックの**いちばん内側**と**自分の関数名**をメモしたか  
5. 再帰なら、基本ケース（止まる枝）と `rest` で短くなっているか  
6. リポジトリ root から `racket code/….rkt` でも再現するか
